# 10_FRONTEND_SPEC.md

## Purpose

本書は、VTuner の frontend 側仕様を定義する。  
frontend は、ユーザーが VTuner を「見て・触って・試して・振り返る」ためのUIを提供する。

本仕様では、frontend の責務、画面構成、各画面の役割、主要UI要素、backend との関係を整理する。  
UI文言は、初期段階では日本語固定とする。

---

## Frontend Role

frontend の主な責務は次のとおり。

- VTuner キャラクターの表示
- 向き切り替え表示（front / side）
- side画像の左右反転表示
- 吹き出し表示
- 音声再生
- 設定フォームUIの表示
- テストUIの提供
- レビューUIの提供
- backend から受信したイベントの描画

frontend は表示・操作・再生に専念し、  
コメント分類や最終発話決定の主責務は持たない。

---

## Screen Structure

frontend は、初期段階では次の4画面で構成する。

1. Preview / Test
2. Basic Settings
3. Review
4. Detailed Rules

# 1. Preview / Test

## Purpose

この画面は、VTuner を実際の配信画面に近い状態で配置し、動かし、反応を確認するための画面である。  
見た目調整とテストは分離せず、同一画面内で扱う。

ユーザーはこの画面で次を行える。

- VTuner の位置調整
- VTuner の向き確認
- VTuner の反転
- VTuner の拡大縮小
- コメント入力テスト
- 単発サンプル再生
- 疑似イベント発火
- 音声確認
- 実際の反応確認
- 画像の仮差し替えと表示確認
- Main Preview 背景の一時差し替え
- 吹き出し位置調整
- 吹き出しサイズの実寸調整

---

## Layout

画面全体は、YouTube風のテスト画面を基本イメージとする。  
ただし、本物そっくりの再現は目的としない。  
YouTube風の雰囲気を持ちつつ、Preview / Test 用として使いやすく整理された画面とする。

画面は、**上段2エリア + 下段1エリア** を基本構成とする。  
できるだけ **スクロールなしで主要要素が見渡せること** を優先する。  
無駄に広い余白を避け、配信画面・コメント履歴・テスト操作が一画面内で把握できる構成とする。

### Header Area

- 画面左上には VTuner ロゴを置く
- VTuner 名は、画面左上の VTuner ロゴ横に表示する
- これは配信画面内の演出表示ではなく、現在編集 / テスト対象となっている VTuner を示す画面ヘッダ情報として扱う
- Main Preview 内への名前表示は初期仕様では必須としない

### Main Preview

- YouTube風の簡易配信画面を表示する
- VTuner はその画面内に重ねて表示する
- 吹き出しもその画面内に表示する
- 配信画面は YouTube風の比率感を維持したまま、Preview / Test 用に縮小して扱えるようにする
- Main Preview は画面内で最も大きい領域とし、見た目確認の主役とする
- `front / side / mirror / 背景 / 立ち絵 / サイズ変更` などの見た目調整UIは、画面外ではなく Main Preview 内に配置する
- VTuner を含む配置対象は、薄い長方形の操作枠として扱う
- 背景は JPEG / PNG で一時差し替えできるようにする
- 背景画像の選択、クリア、表示確認を Main Preview 内で行えるようにする

### Bubble Edit Frame

- 吹き出し本体を直接リサイズ対象にはしない
- 吹き出しの外側に、薄い長方形の編集枠を持つ
- 位置移動および拡大縮小は、この編集枠に対して行う
- 吹き出し本体は自然な見た目を保ち、編集UI由来の四角い端や内部スクロールを見せないことを優先する
- VTuner の編集枠と同系統の見え方に揃える

### Comment / Test History

- コメント欄はダミー表示ではなく、ユーザーがこの画面で行ったテストの履歴を表示する
- 手入力コメント、単発サンプル再生、疑似イベント発火の結果を履歴として蓄積する
- VTuner の返答も履歴として確認できるようにする
- 単なる飾りのコメントは置かない
- 履歴表示を上部、コメント入力を下部に配置する

### VTuner Placement

- VTuner は画面内で自由にドラッグ移動できる
- 初期位置は右下とする
- 配信内容に応じて、ユーザーが配置を変更できるようにする
- 配置結果は appearance 設定へ反映できるようにする
- 移動対象はキャラクター単体ではなく、長方形の操作枠ごととする

### Bottom Test Area

Bottom Area は、説明を読むための情報置き場ではなく、  
通常コメント以外の条件や挙動を実際に試すためのテスト領域とする。

ここでは、次のようなテストを行えるようにする。

- 無コメント状態の継続テスト
- 連投時の取捨選択テスト
- NGワード無視テスト
- 条件イベント発火テスト
- 発火した反応種別の確認
- 無視 / 保留 / 不採用結果の確認

Bottom Area は、入力内容を重複表示したり、音声状態を冗長に説明する場所にはしない。

## VTuner Preview Behavior

### Display

- front / side の表示を行う
- side は mirror を使って左右反転表示する
- back は初期仕様では扱わない

### Placement

- VTuner は Main Preview 上でドラッグ移動できる
- 初期位置は右下とする
- ドラッグ結果は appearance 設定へ反映できるようにする
- 移動対象はキャラクター単体ではなく、長方形の操作枠ごととする

### Size

- 初期段階ではサイズ調整を許可する
- 配信中の既存VTuber的な操作感に近い形で扱えることを優先する
- `+ / -` のような倍率操作は初期仕様では採用しない
- 長方形の操作枠の端を掴んで拡大縮小する方式を想定する
- 調整UIの形は後で詳細化するが、まずは分かりやすく操作できることを優先する

### Orientation / Mirror

- front / side の切り替えを行える
- 必要に応じて左右反転を行える
- 初期段階では切り替えは瞬時でよい
- 補間アニメーションは必須ではない
- コメント読上げ時は side、配信者向け発言時は front のような挙動を、Preview / Test 上で確認できるようにする
- 向き制御ロジックの実装責務はバックエンド側でもよいが、Preview / Test では結果を視覚的に確認できることを優先する

### Bubble Placement

- 吹き出しは Preview / Test 上で直接位置調整できるようにする
- Main Preview 上でドラッグ移動できることを優先する
- 調整結果は実配信時の表示位置として扱えるようにする

### Bubble Resize

- 吹き出しサイズは Preview / Test 上で直接調整できるようにしてよい
- 角を掴んで大きさを変更する方式を想定する
- 調整結果は Basic Settings の表示設定に反映できるようにする

### Preview Background

- Main Preview の背景は JPEG / PNG で一時差し替えできるようにする
- 背景画像は Preview / Test 上で見え方確認に使う
- 背景変更UIは Main Preview 内に配置する
- 必要に応じて実運用設定へ反映できるようにする

## Interactive Elements

### Comment Test Input

- ユーザーが自由にコメント文を入力できる
- 必要に応じて投稿者名入力もできる
- 送信すると backend にテストコメントイベントを送る
- 結果は Comment / Test History と Main Preview に反映される

### Single Sample Replay

- あらかじめ用意したサンプルコメントの中から 1 件を選択して再生できる
- 単発サンプルはランダム実行ではなく、ユーザーが選択してから実行する方式を優先する
- 単発サンプル実行UIは、サンプル選択UIとセットで成立するものとする
- 再生結果は Comment / Test History と Main Preview に反映される

### Synthetic Event Trigger

実時間では試しづらいイベントは、手動ボタンで疑似的に発火できるようにする。

初期候補:

- 無言イベント発火
- 初回名前イベント発火
- 回数到達イベント発火
- 話題振りイベント発火
- 特定ユーザー来訪イベント発火

### Condition Test Area

Bottom Test Area では、通常コメント以外の条件テストを行えるようにする。

初期候補:

- 無コメント
- 連投
- NGワードコメント
- センシティブ除外コメント
- 長すぎる名前
- 読みにくい名前
- 無視対象コメント

各条件は固定ボタンとして個別に並べるのではなく、  
条件値やオプションを持てるテストプリセットとして扱うことを優先する。

### Voice Test

- 現在の音声設定でサンプル再生できる
- テスト反応時にも音声確認できる

### Temporary Asset Import

- front画像 / side画像を一時的に差し替えて見た目を試せる
- ドラッグ＆ドロップまたはファイル選択を許可する
- 問題なければ Basic Settings 側へ反映できるようにする
- 立ち絵差し替えUIは Main Preview 内に配置する

### Reset

- この画面には、現在のテスト状態を破棄してプリセットに戻す単一ボタンを置く
- 個別リセットよりも、全体を既定状態へ戻すことを優先する

---

## Bottom Test Area

Bottom Test Area は、条件テストを個別ボタンで直接発火する場所ではなく、  
条件を持ったテストプリセットを作成・保存・再実行するための領域とする。

各テストは、単発操作ではなく、  
条件・オプション・確認対象をまとめたひとつのテスト行として扱う。

### Basic Structure

各テストプリセットは次の情報を持つ。

- テスト名
- 条件種別
- 条件値
- オプション
- 確認対象
- 実行ボタン

### Example Conditions

- 無コメント
  - 秒数指定
  - 5秒前カウント有無

- 連投
  - 件数指定
  - 同一ユーザー / 複数ユーザー
  - 間隔指定

- コメント条件
  - NGワード含有
  - センシティブ除外
  - 長すぎる名前
  - 読みにくい名前
  - 無視対象コメント

- 条件イベント
  - 無言イベント
  - 初回名前イベント
  - 回数到達イベント
  - 話題振りイベント
  - 特定ユーザー来訪イベント

### Preset Operations

- 新規作成
- 保存
- 複製
- 削除
- 実行

### Result Display

結果は全体の共通ログにまとめるのではなく、  
実行したテストプリセットごとに最小限表示する。

表示粒度は次にとどめる。

- 実行結果
  - 発火
  - 無視
  - 保留
  - 不採用
  - エラー

- 短い理由ラベル

- 使用反応名

### UI Policy

- 30秒、60秒、120秒などを固定ボタンで並べない
- 条件値は選択または数値入力で持てるようにする
- テストは再利用可能なプリセットとして保存できるようにする
- Bottom Area は説明表示よりも実行性を優先する

---

## Initial Constraints

- 本物そっくりの YouTube UI を再現する必要はない
- YouTube風の雰囲気を持つ簡易画面でよい
- コメント・反応・配置確認が分かりやすいことを優先する
- 画面内での VTuner 配置の触りやすさを重視する
- スクロールなしで主要要素を確認しやすい構成を優先する
- コメント欄はテスト履歴として扱う
- UI文言は日本語固定とする

# 2. Basic Settings

## Purpose

この画面は、VTuner の基本設定を行う場所である。  
普段の利用で必要な設定は、この画面だけで完結できることを優先する。

この画面では、JSON やブラウザAIを主役にせず、  
フォームUIで直接設定できることを重視する。

Basic Settings では、キャラ全体に共通して効く設定を扱う。  
条件ごと・ルールごとに変わるものは Detailed Rules で扱う。

また、Basic Settings は「追加する場所」ではなく、  
**既に存在する定義から選ぶ場所** として扱う。  
新しい定義の追加は Detailed Rules 側で行う。

ただし、**フォント追加** のような表示専用の軽量な追加は Basic Settings 側で扱ってよい。

---

## Section Order

画面2は次の順でセクションを持つ。

1. Basic Profile
2. Character Profile
3. Character Style
4. Speech Style
5. Voice

## Display Language Policy

- ユーザー向けのUI文言は日本語固定とする
- 内部カテゴリ名や内部キー名を、そのまま主UIに露出しない
- 内部名が必要な箇所では補助表示してもよいが、主表示は日本語を優先する

## 2.1 Basic Profile

### Purpose
仲介者キャラクターの基本情報を定義する。

### Main Fields
- VTuner名
- 一人称
- 視聴者の呼び方
- 配信者の呼び方

### UI Policy
- VTuner名は自由入力
- 一人称は自由入力
- 呼び方も基本は自由入力

### Notes
- VTuner名はキャラクターの存在感を作るために必要
- 一人称は選択式に固定しない
- 呼び方はテンプレート文の差し込みにも使う

---

## 2.2 Character Profile

### Purpose
キャラクターのプロフィール情報を定義する。

### Main Fields
- 年齢
- 身長
- 体重
- 誕生日
- 好きなもの
- 苦手なもの
- 備考
- 公開 / 非公開メモ

### UI Policy
- プリセット入力でも自由入力でもよい
- 項目を完全固定しすぎず、追記しやすい構造が望ましい

### Notes
- Character Profile はキャラの世界観を作るための素材である
- ここで設定した内容を、そのまま常に話すわけではない
- 「体重はごまかす」などの返答ルールは Detailed Rules 側で扱う

---

## 2.3 Character Style

### Purpose
キャラクターの大まかな性格方針を決める。

### Main Fields
- 性格タイプ
- 視聴者向け方針
- 配信者向け方針

### Initial Preset Count
- 初期の性格タイプは 4種では少ない
- 初期段階でも 8〜12種程度を用意することを推奨する

### Preset Examples
初期候補例:
- 丁寧な案内役
- やわらかい相棒
- 観察者タイプ
- 少し毒舌な補佐役
- 元気な盛り上げ役
- クールな秘書役
- 世話焼きなお姉さん役
- いたずらっぽい相棒
- 上品なお嬢様風
- 中性的で静かな補助役

### UI Policy
- 連続値のバーは使わない
- 離散的な選択式を基本とする
- `＋追加` は Basic Settings 側には置かない

### Notes
- 性格タイプは、発話を直接生成するためのものではない
- 性格タイプは、初期ルールや初期候補文の方向性を決めるために使う
- 視聴者向け / 配信者向けで方針を分けられることが重要
- 新しい性格タイプ定義の追加は Detailed Rules 側で行い、その結果が Basic Settings の選択肢に反映される

---

## 2.4 Speech Style

### Purpose
具体的な話し方・口調・言い回しの方針を定義する。

### Main Fields
- 口調タイプ
- 語尾タイプ
- 好きな言い回し
- 避けたい言い回し
- 禁止表現

### UI Policy
- 基本は選択式を優先する
- 自由入力を置きすぎない
- ここは直接セリフを作る場所ではない
- `＋追加` は Basic Settings 側には置かない

### Notes
- 「話し方」は、テンプレートの選び方と補正方針を決める場所とする
- 実際の反応テンプレートは Detailed Rules で扱う
- 話し方だけでVTunerの全発話が決まる設計にはしない
- 新しい口調タイプや語尾タイプの定義追加は Detailed Rules 側で行い、その結果が Basic Settings の選択肢に反映される

---

## 2.5 Voice

### Purpose
キャラ全体に共通する音声設定を行う。

### Main Fields
- 音声を使う / 使わない
- 声の種類
- 話すスピード
- 音量
- テスト再生

### Voice Control Policy
- 話すスピードは 5段階固定の選択式とする
- 例:
  - とてもゆっくり
  - ゆっくり
  - 標準
  - やや速い
  - 速い
- 音量は連続的な調整価値が高いため、スライダーでよい

### Notes
- Voice はキャラ全体に共通して効く設定として扱う
- ルールごとに声を変える構造は初期段階では採用しない
- 音声の細かな条件分岐は初期仕様では扱わない

---

## 2.6 Display / Assets

### Purpose
VTuner の見た目に関する基本設定を行う。

### Main Fields
- 使用イラスト
- front画像
- side画像
- 初期表示位置
- サイズ
- 反転設定

### Asset Import Policy
- front画像 / side画像は Basic Settings でも登録できる
- ドラッグ＆ドロップまたはファイル選択を許可してよい
- 大きな `＋` を押してインポートするような構造は有効
- Preview / Test でも仮差し替えはできるが、正式な登録先は Basic Settings とする

### Notes
- side は mirror 利用を前提とする
- back は初期仕様では扱わない
- Preview / Test は見た目調整の場、Basic Settings は正式登録の場として役割を分ける

## 2.7 Speech Bubble / Typography

### Purpose
吹き出しや文字の見た目を、キャラ全体に共通する表示設定として管理する。

### Main Fields
- 吹き出しを使う / 使わない
- 吹き出しの形
- フォント
- 文字サイズ
- 文字色
- 背景色
- 枠線の有無

### Typography Policy
- フォントは選択式とする
- 文字サイズは 5段階固定の選択式とする
  - 小
  - やや小
  - 標準
  - やや大
  - 大
- 文字色は少数の基本パレットから選ぶ方式とする
- 背景色も少数の基本パレットから選ぶ方式とする
- 自由なカラーピッカーは初期段階では必須ではない

### Bubble Size Policy
- 吹き出しサイズはフォーム項目にしない
- 吹き出しサイズは Preview / Test 上で直接調整する
- 角を掴んで大きさを変更する方式を想定する
- 調整結果は表示設定へ保存できるようにする

### Font Import Policy
- フォント追加は Basic Settings 側で扱ってよい
- ローカル用フォントをインポートし、選択肢へ追加する構造を許可する
- フォント追加は表示専用の軽量設定として扱い、Detailed Rules へは送らない

### Notes
- 吹き出しや文字の見た目は、キャラ全体に共通する表示設定として扱う
- ルールごとに吹き出しやフォントを変える構造は初期仕様では採用しない
- 読みやすさを優先し、過度な装飾は避ける

## 2.8 Reaction Categories

### Purpose
どの種類の反応を使うかの大枠だけを決める。

### Main Fields
- 挨拶
- 褒め
- 質問
- 応援
- 共感
- 笑い
- 話題振り
- 初見反応
- 常連反応
- 未分類

### UI Policy
- ここではカテゴリ単位の有効 / 無効だけを扱う
- `＋追加` は Basic Settings 側には置かない

### Notes
- 実際のキーワード、条件、候補文、対象は Detailed Rules で扱う
- 新しい反応カテゴリ定義の追加は Detailed Rules 側で行い、その結果が Basic Settings の選択肢やカテゴリ一覧に反映される
- Basic Settings は「どんな子か」と「どの種類の反応を使うか」を決める場所であり、「いつ何を話すか」を細かく決める場所ではない

# 3. Review

## Purpose

この画面は、配信後に VTuner の見逃しや改善点を回収する場所である。  
自動学習ではなく、ユーザーが意味を仕分けして、必要なものだけを追加する。

ただし、この画面は本格的なルール編集の場ではない。  
配信後レビューでは、コメントを見直して仕分けし、  
必要なものを patch 候補や追加候補として整理することを主目的とする。

正式なルール追加や詳細な編集は Detailed Rules 側で行う。

---

## Main Responsibilities

- コメント一覧表示
- 状態別表示
- `unknown` の確認
- ignore
- 既存カテゴリへの追加候補化
- 新カテゴリ候補のストック
- patch 候補整理

---

## Main Display Targets

### Comment State
初期対象:
- displayed
- skipped
- ignored
- unknown

### Main Operations
- `unknown` だけ絞り込む
- 既存カテゴリへの追加候補にする
- ignore にする
- 新カテゴリ候補として保留する

### UI Language Policy
- 主表示は日本語とする
- 状態ラベルも、必要に応じて「表示済み」「未採用」「無視」「未分類」などの日本語を併記する
- 画面上の操作文言は、即時確定に見えすぎない表現を優先する

---

## Review Result Handling

- review で行った判断は、原則としてその場で本体設定へ全面反映しない
- review 結果は、まず patch 候補または追加候補として保持する
- `既存カテゴリへの追加` は、完成済みルールへ直接書き込むことではなく、既存カテゴリへの追加候補として扱う
- `新カテゴリ候補として保留` は、Detailed Rules 側で後から正式整理する前提で保持する
- review は「仕分けと候補化」、Detailed Rules は「正式追加と詳細編集」という役割分担を維持する

---

## Initial UX Direction

- 全件を細かく処理させない
- まず `unknown` を中心に扱う
- 一括操作しやすいことを優先する
- review は仕分けと候補化を優先し、その場で複雑な定義編集をさせない
- 詳細クラスタリングは後回しでよい

---

# 4. Detailed Rules

## Purpose

この画面は、「いつ、誰に、何を話すか」を細かく決める場所である。  
通常反応・条件イベント・特定ユーザー対応など、実際の発話ルールはこの画面で扱う。

JSON やブラウザAIは、この画面でのみ前面に出す。  
ただし主役はあくまでルール一覧とルール編集である。

また、Detailed Rules はルール追加だけでなく、  
**新しい定義の追加** も扱う。  
ここで追加された定義は、Basic Settings 側の選択肢に反映される。

---

## Role Separation

- Basic Settings: キャラの基本方針を決める
- Detailed Rules: 具体的な発話ルールと追加定義を決める

この分離により、基本設定画面の破綻を防ぐ。

---

## Relationship with Basic Settings

- Basic Settings で決めた性格タイプや話し方タイプに応じて、Detailed Rules の初期プリセットを変えてよい
- Detailed Rules ではユーザーが個別に編集できることを優先する
- Basic Settings が更新されても、既に編集済みの詳細ルールを自動破壊しない
- Detailed Rules で追加した定義は、Basic Settings の選択肢一覧に反映される
- ただし、フォント追加や吹き出しの見た目調整は Basic Settings 側で扱う

---

# Frontend State Responsibilities

frontend が持つべき基本状態は次のとおり。

- 現在の画面
- VTuner の表示状態
- 現在のプレビュー位置
- 現在のプレビューサイズ
- 現在の反転状態
- 現在の吹き出しサイズ
- 受信済み message queue
- テスト入力内容
- 音声設定状態
- レビュー画面の絞り込み状態
- 詳細ルール一覧の表示状態
- ルールごとの有効 / 無効状態
- ルール追加方法の選択状態
- 追加定義の反映状態
- フォント選択肢一覧

---

# Frontend and Backend Boundary

## frontend が担当するもの
- 描画
- フォーム
- ドラッグ操作
- 音声再生
- テスト操作
- レビュー表示
- 基本設定UI
- 詳細ルールUI
- 表示設定の保存
- フォント追加UI

## backend が担当するもの
- コメント分類
- 発話決定
- target / mode / emotion / displayText の決定
- runtime state
- review 用ログ生成
- patch の適用ロジック
- 条件イベントの判定
- JSON読込内容の検証と反映
- 自然言語イベント記述の内部ルール化
- 追加定義の保存と選択肢への反映

---

# Initial Non-Goals

初期の frontend 仕様では、次を急いで扱わない。

- 高度なアニメーション
- 複数オーバーレイの同時制御
- 複雑なレスポンシブ調整
- 本物そっくりのYouTube UI再現
- 高度なUIテーマ切り替え
- review 画面の自動クラスタリング
- 未知のJSONから未知のUIを自動生成すること
- 性格バーとレーダーによる細かい連続値制御
- 条件ごとに吹き出し見た目を切り替えること

---

## Summary

frontend は、VTuner を「見て・触って・試して・振り返る」ための場である。  
初期段階では 4画面構成とし、

1. Preview / Test
2. Basic Settings
3. Review
4. Detailed Rules

を中心に組み立てる。

特に Basic Settings では、
- VTuner名
- 一人称
- 呼び方
- キャラプロフィール
- 性格タイプ
- 話し方タイプ
- 音声
- 使用イラスト
- 吹き出しと文字
- 反応カテゴリ

を扱い、  
Detailed Rules では、
- いつ
- 誰に
- 何を話すか
- 新しい定義をどう追加するか

を扱う。
