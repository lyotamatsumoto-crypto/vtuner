# 18_DETAILED_RULES_SCREEN_SPEC.md

## Purpose

本書は、VTuner における Detailed Rules 画面の仕様を定義する。

Detailed Rules 画面は、通常反応・条件イベント・特定ユーザー対応など、  
VTuner が「いつ、誰に、何を話すか」を具体的に定義・編集するための専用画面である。

本書の目的は次のとおり。

- 発話ルールの中心画面を定義する
- ルール管理とルール追加を整理する
- JSON の役割を明確にする
- ブラウザAI用の指示文の意味を明確にする
- 追加済みルールを一覧で管理できるようにする
- 追加済み定義が Basic Settings にどう反映されるかを明確にする

---

## Basic Policy

- Basic Settings はキャラの基本方針を決める場とする
- Detailed Rules は具体的な発話ルールを決める場とする
- JSON は通常設定の主役ではない
- JSON はルールや定義をまとめて追加・共有するための手段とする
- ブラウザAIは「設定の代わり」ではなく、「追加ルール案・追加定義案を作る補助」とする
- まずルール一覧を主役にし、その次に追加方法を見せる
- ルールと発話文を分離しすぎない
- 各ルールの中で、条件・対象・候補文がひとまとまりで見えるようにする
- Detailed Rules で追加した定義は、Basic Settings の選択肢に反映される
- 吹き出し見た目、文字色、背景色、文字サイズ、フォント追加のような**表示共通設定**は Detailed Rules ではなく Basic Settings 側で扱う

---

## UI Language Policy

- UI文言は日本語固定とする
- `Behavior`, `Advanced Event`, `Template Block` などの内部用語は主UIに出さない
- `設定生成補助 (Behavior)` のような名称は使わない
- 代わりに「ブラウザAI用の指示文を作る」「追加ルールJSONを読み込む」など、機能が分かる日本語を使う
- `last_comment_age > 180s` のような内部条件式は主UIに出さない
- 追加対象や追加方法が曖昧なまま、JSONやAIの入口を見せない

---

## Screen Role

Detailed Rules 画面は、次の役割を持つ。

- 既存ルールの一覧表示
- ルールの編集
- 条件の編集
- 発話相手の編集
- 候補文の編集
- 有効 / 無効の切り替え
- 削除
- 新しいルールの追加
- 新しい定義の追加
- JSONからの追加
- ブラウザAI用指示文の作成

Detailed Rules 画面は、**Basic Settings では扱わない可変ルールと拡張定義を管理する場**である。  
見た目の共通設定を増やしすぎないための隔離でもある。

---

## Screen Structure

Detailed Rules 画面は、初期段階では次の構成を持つ。

1. ルール一覧
2. ルール編集
3. ルール追加
4. 定義追加
5. 指示文 / JSON 操作

---

# 1. ルール一覧

## Purpose

現在登録されている発話ルールを一覧で管理する。  
この画面の主役であり、まず最初に見えるべき領域である。

## Initial Rule Groups

- 通常反応
- 条件イベント
- 特定ユーザーイベント
- 名前イベント
- 時間イベント

## Per-Rule Display

各ルールカードには、最低限次を表示する。

- ルール名
- ルール種別
- 有効 / 無効
- 条件の要約
- 発話相手
- 候補文の件数
- 更新元
  - 初期プリセット
  - 手動追加
  - JSON読込
  - ブラウザAI案

## Per-Rule Actions

- 編集
- 有効化 / 無効化
- 削除
- 複製

## Notes

- 一覧はカード形式が望ましい
- 「いつ使うルールか」が見えることが重要
- 候補文だけが独立して並ぶ構造にしない
- ルールと候補文は、必ず同じ文脈で見えるようにする
- 一覧を見ただけで「この条件でこの相手にこの系統の文を使う」が読めることを優先する

---

# 2. ルール編集

## Purpose

各ルールの中身を直接編集する。

## Per-Rule Editable Elements

- 条件
- 発話相手
- 候補文
- ON / OFF
- 補足メモ

## Condition Editing Policy

ユーザー向けUIでは、内部条件式を直接入力させない。  
たとえば次のような形にする。

- コメントがない時間: 180秒
- 誰に向けて話すか: 配信者
- 発言タイプ: 話題を振る

または

- 対象ユーザー名: たろう
- 発言相手: 視聴者
- 反応種別: 特別な挨拶

## Candidate Sentence Policy

候補文は、各ルールの中に属するものとして編集する。

例:
- ルール: コメントが180秒ない
- 候補文A: 少し静かですね
- 候補文B: 何か話題を出してみましょうか

## Notes

- 「候補テンプレートの選択」を独立セクションに分けない
- 各ルールの中で、候補文を管理する
- 候補文は、見るだけでなく編集できる必要がある
- ルールを見たときに、「この条件でこの文を話す」が自然に読み取れる構造を優先する
- 候補文はルールにぶら下がるものとして表示し、Basic Settings には持ち込まない

---

# 3. ルール追加

## Purpose

新しい発話ルールを増やす方法を提供する。

## Initial Add Methods

- 手動で追加する
- プリセットから追加する
- JSONから追加する
- ブラウザAI用の指示文を作る

## Add Target Types

- 通常反応
- 条件イベント
- 特定ユーザーイベント
- 名前イベント
- 時間イベント

## Notes

- まず何を追加したいかを選ばせる
- その次に、どう追加するかを選ばせる
- 「何に対するJSONなのか」が曖昧な状態で読込ボタンを見せない
- 追加したルールは、一覧にすぐ反映される必要がある
- Basic Settings で見えているカテゴリON/OFFと、ルール追加の実体がずれないようにする

---

# 4. 定義追加

## Purpose

Basic Settings 側で選択肢として使われる「定義」を増やす。

ここでいう定義とは、たとえば次のようなものである。

- 性格タイプ
- 口調タイプ
- 語尾タイプ
- 反応カテゴリ

## Initial Definition Types

- 性格タイプ定義
- 口調タイプ定義
- 語尾タイプ定義
- 反応カテゴリ定義

## Definition Role

定義は、Basic Settings 側の選択肢を増やすためのものである。  
ただし、名前だけ増やしても意味がないため、必要に応じて補足情報や関連候補を持てるようにする。

例:
- 性格タイプ名
- 簡単な説明
- 関連する初期ルールの方向性
- 関連する候補文の方向性

## Definition Reflection Policy

Detailed Rules で追加した定義は、保存後に Basic Settings の選択肢へ反映する。

例:
- 新しい性格タイプ定義を追加した  
  → Basic Settings の「性格タイプ」選択肢に表示される
- 新しい口調タイプ定義を追加した  
  → Basic Settings の「口調タイプ」選択肢に表示される
- 新しい反応カテゴリ定義を追加した  
  → Basic Settings の「反応カテゴリ」一覧に表示される

## Notes

- Basic Settings 側には `＋追加` を置かない
- 定義の追加は Detailed Rules 側で行う
- Detailed Rules で追加した定義は、Basic Settings の選択肢に反映される
- 反映後、Basic Settings は「追加済み定義から選ぶ場所」として機能する
- フォントはここで扱わない
- 吹き出し形状や文字色パレットもここで扱わない

---

# 5. 指示文 / JSON 操作

## Purpose

ルール追加や定義追加の補助手段として、ブラウザAI用の指示文作成と JSON 読込を行う。

---

## 5.1 ブラウザAI用の指示文を作る

### Purpose
ブラウザAIに、指定した種類の追加ルール案または追加定義案を作ってもらうための文章を生成する。

### UI Naming
次のような名称を使う。

- ブラウザAI用の指示文を作る
- 外部AIに渡す文章を作る
- 追加案を作るための指示文

### Notes
- 「学習」「自動生成エンジン」のような誤解を招く表現は使わない
- 指示文は、選んだ追加対象に対応した内容である必要がある
- たとえば「特定ユーザーイベントを追加」を選んだなら、そのための指示文を出す
- たとえば「新しい口調タイプを追加」を選んだなら、そのための指示文を出す
- 指示文は「何を追加したいか」が明確な状態でしか出さない

---

## 5.2 追加ルールJSONを読み込む

### Purpose
追加対象に応じた JSON を読み込み、ルール一覧または定義一覧に追加する。

### UI Naming
次のような名称を使う。

- 追加ルールJSONを読み込む
- この種類の追加ルールを読み込む
- ルールJSONを取り込む
- 定義JSONを取り込む

### Notes
- 「何に対してのJSONか」が分からない状態で読込ボタンを出さない
- ルール追加か定義追加かを先に選ばせる
- 読み込んだ結果は、一覧に追加された形で見える必要がある
- JSON読込結果が一覧に反映されて初めて意味を持つ

---

## 5.3 JSONの役割

### Positioning
JSONは、通常設定の保存先そのものではなく、  
主に「追加ルール」または「追加定義」をまとめて増やすための形式として扱う。

### Notes
- ユーザーが普段触るものではない
- UIにない複雑なルールを増やしたいときに使う
- 大量の候補文やイベントをまとめて追加したいときに使う
- 新しい性格タイプや反応カテゴリなどの定義を増やしたいときにも使う
- 共有やバックアップにも向く
- フォント追加や色パレット追加のような表示系設定は、初期段階では JSON 拡張対象にしない

---

## Relationship with Basic Settings

- Basic Settings で決めた性格タイプや話し方タイプは、初期ルール案の方向性に影響してよい
- ただし、Detailed Rules は独立して編集可能であるべき
- Basic Settings を直したからといって、既存の詳細ルールを自動上書きしない
- Detailed Rules で追加した定義は、Basic Settings の選択肢に反映される
- Basic Settings は「選ぶ場所」、Detailed Rules は「増やす場所」として役割を分ける
- フォント追加、文字色、背景色、吹き出し形状、吹き出しサイズは Basic Settings / Preview 側に残す

---

## Natural Language Input

## Purpose

将来的には、自然言語からルール案や定義案を作る補助導線を持てるようにする。

例:
- 「たろうさんが来たら『いらっしゃい、待ってたよ』と返す」
- 「5分誰も来なかったら、配信者に話題を振る」
- 「少し上品で距離感のある口調タイプを追加したい」
- 「視聴者をブラザーと呼ぶ反応カテゴリを追加したい」

## Notes

- 自然言語はそのまま実行しない
- backend 側で内部ルールまたは内部定義に変換する前提とする
- 変換結果はルール一覧または定義一覧に追加される形で見えるようにする
- 変換後の内容をユーザーが確認・編集できることを優先する

---

## Validation Policy

- JSON読込時は妥当性検証を行う
- 読み込めない場合は、何が不足しているか分かるようにする
- 読込成功時は、どの種類のルールまたは定義として何件追加されたかを明示する
- 読込結果が一覧に反映されることが重要である
- Basic Settings 側へ反映される定義については、重複や名前衝突にも注意する

---

## Relationship with Other Screens

### With Preview / Test
- 追加した詳細ルールは Preview / Test 画面で検証できる必要がある
- 条件イベントや特定ユーザーイベントは、疑似発火ボタンで試せることが望ましい

### With Review
- Review で見つけた追加候補を、Detailed Rules 画面に送って新ルール化できると望ましい

### With Basic Settings
- Detailed Rules で追加した定義は Basic Settings の選択肢に反映される
- Basic Settings では、その追加済み定義を選べるようにする
- Basic Settings 側で見た目設定を増やしすぎないため、拡張定義は Detailed Rules 側に寄せる

---

## Initial Non-Goals

初期段階では、次を急いで扱わない。

- 未知のJSONから未知のUIを自動生成すること
- JSONだけで全設定を置き換えること
- AIが勝手に詳細ルールや定義を確定すること
- 完全自動の自然言語解釈
- 連続値の性格パラメータを細かく反映すること
- フォントや色設定のJSON拡張
- 条件ごとに吹き出し見た目を切り替えること

---

## Summary

Detailed Rules 画面は、VTuner が「いつ、誰に、何を話すか」を具体的に決めるための専用画面である。

この画面では、

1. ルール一覧を管理する
2. 各ルールの条件・相手・候補文を編集する
3. 必要なら新しいルールを追加する
4. 必要なら新しい定義を追加する
5. JSONやブラウザAIは、その追加手段として使う

という流れを明確にする。

これにより、JSONやブラウザAIが「意味のない窓口」になることを防ぎ、  
Basic Settings 画面の破綻も防ぐ。
