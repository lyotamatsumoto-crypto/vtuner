# AI_JSON_STUDIO_SPEC.md

## Purpose

本書は、AI / JSON Studio の目的、生成対象、UI 構造、外部ブラウザ AI ワークフロー、JSON 検証、再修正プロンプト生成、履歴、マイプリセット化を定義する文書である。

AI / JSON Studio は、アプリ内部に AI を持たない前提のもとで、外部ブラウザ AI を使った設定生成を支援する専用画面である。

---

## Core Principle

- アプリ内部に AI は存在しない
- AI / JSON Studio は外部ブラウザ AI 用の支援画面である
- 自然文入力から外部 AI 用プロンプトを生成する
- 返ってきた JSON を検証する
- 検証成功なら基本本採用する
- 検証失敗なら再修正プロンプトを生成する
- ユーザーが JSON を直接編集することを前提にしない

---

## Main Purpose

- 非プログラマでも JSON 形式の設定や定義を作成できるようにする
- 生成対象ごとに入力支援を行う
- JSON を壊れにくいワークフローで扱えるようにする
- エラー時の詰まりを減らす
- 採用履歴と修正履歴を追えるようにする
- マイプリセット化へつなぐ

---

## Generation Target Tabs

AI / JSON Studio の生成対象タブは次のとおり。

- 人格
- 返答カテゴリ
- 返答集
- 条件イベント
- カテゴリ定義
- エラー修正

## UI Structure

### Overall structure
- 左右 2 カラム常設型
- ただし生成対象ごとに専用作業面を切り替える

### Left column
- 生成対象の説明
- かんたん入力
- 詳細入力
- 関連プリセット選択
- プロンプト生成
- プロンプトコピー

### Right column
- JSON 貼り付け / 読込
- 検証結果
- エラー内容
- 再修正プロンプト
- 差分要約
- 採用
- 履歴
- マイプリセット化導線

---

## Input Modes

### かんたん入力
短文で方向性を入れる。

例:
- 性格
- 雰囲気
- 禁止したいこと
- 相手との距離感

### 詳細入力
より細かく入力する。

例:
- 配信者向け態度
- 視聴者向け態度
- 好きな言い回し
- 避けたい言い回し
- カテゴリ方向性
- 返答例の方向性

---

## Generation Targets

### 人格
作りたいもの:
- 基本性格
- 視聴者向け態度
- 配信者向け態度
- 禁止表現
- 口調
- 語尾
- 好きな言い回し

### 返答カテゴリ
作りたいもの:
- カテゴリ名
- 説明
- 対象例
- 差分説明

### 返答集
作りたいもの:
- カテゴリごとの候補文集
- 対象別差分
- 温度感
- 禁止表現との整合

### 条件イベント
作りたいもの:
- 種別
- 条件
- 発話対象
- 使用返答または返答方向性
- 補助メモ

### カテゴリ定義
作りたいもの:
- 新カテゴリ名
- 説明
- 対象例
- 既存カテゴリとの差

### エラー修正
作りたいもの:
- 元 JSON の不足や不整合を修正させるための再依頼用プロンプト

## External AI Workflow

1. ユーザーが生成対象タブを選ぶ
2. かんたん入力または詳細入力を行う
3. VTuner がルールベースのプロンプトを生成する
4. ユーザーがそのプロンプトをブラウザ AI に貼る
5. ブラウザ AI が JSON を返す
6. ユーザーが JSON を貼る / 読み込む
7. VTuner が検証する
8. 検証成功なら差分要約を表示する
9. ユーザーが採用する
10. Adopted Changes に入る
11. 必要ならマイプリセット化できる

---

## Validation Policy

- 生成対象ごとに検証する
- 想定外キーを許可しない
- 必須キー不足を許可しない
- 型不一致を許可しない
- enum 不一致を許可しない
- 検証成功時は差分要約を表示する
- 検証失敗時はエラー内容を表示する

---

## Correction Prompt Policy

検証失敗時は、エラー内容に応じた再修正プロンプトを生成する。

再修正プロンプトには、少なくとも次を含める。

- 何の JSON を修正してほしいか
- どのキーが不足しているか
- どの型が違うか
- どの値が規格外か
- 余計なキーがあるか
- 余計な説明文をつけず JSON のみを返すこと

---

## Adoption Policy

- JSON 検証成功なら基本本採用とする
- ただし採用前に差分要約は見せる
- JSON 全文手編集を前提にしない
- 採用後は Adopted Changes に入れる
- compile によって本体へ反映される

---

## History Policy

履歴は少なくとも次の 3 種を持つ。

### 生成履歴
- 生成日時
- 生成対象
- 元自然文
- 使用テンプレート名
- 対象キャラ名
- 性格軸

### 採用履歴
- 採用日時
- 採用対象
- 採用名
- どの生成履歴から来たか

### 修正履歴
- エラー内容
- 再修正プロンプト
- 修正版 JSON
- 成功までの試行回数

---

## Naming Policy

履歴名や候補名の推奨形式:

`キャラ名__対象__用途__性格軸__日時`

例:
- ヴィヴィ__viewer__reply-pack__ツンデレ__2026-04-18-2230
- ヴィヴィ__streamer__topic-nudge__クーデレ__2026-04-18-2242

---

## My Preset Promotion

次の条件を満たした場合、マイプリセットへ昇格できる。

- 検証成功
- 採用済み
- 名前付け済み

マイプリセットへ昇格できる候補:
- 人格
- 返答カテゴリ
- 返答集
- 条件イベント
- カテゴリ定義
- 派生編集済みの JSON 生成物

---

## Boundary with Detailed Rules

AI / JSON Studio は、生成、検証、再修正、採用を担う。  
Detailed Rules は、採用済みデータの正式編集を担う。

AI / JSON Studio に置くもの:
- 生成
- JSON 取込
- JSON 検証
- エラー修正プロンプト
- 採用
- 履歴
- マイプリセット化

Detailed Rules に置くもの:
- 採用済みルール編集
- 採用済み定義編集
- 条件編集
- 候補文編集
- 手動追加
- patch 採用

## Notes

- AI / JSON Studio は一本道ウィザードではなく、対象別の常設作業面とする
- 基本機能よりも深い生成支援のための画面として扱う
- ただし非プログラマでも詰まらないよう、検証とエラー修正導線を必須とする
